# Prior art & key findings

The research that shaped the design. Each finding maps to a concrete architectural decision.

## 1. Agents are a new class of end-user

The SWE-agent paper (Yang et al., NeurIPS 2024, arXiv:2405.15793) introduced the Agent-Computer Interface concept: careful interface design — not model changes — drove a 3× improvement on SWE-bench. Its principles: actions should be **simple** (few options, concise docs), **efficient** (consolidate high-value operations), and **guarded** (its edit command runs a linter and refuses invalid edits). Also: "all content has a fixed cost in memory and computation for LMs and distracting context can harm performance," and LMs "lack the visual understanding abilities to directly operate GUI-based applications."

→ DocxEngine: ~16 consolidated tools, hash-guarded edits, projection instead of raw XML, render-preview for the missing "eyes."

## 2. Anthropic's tool-design guidance

"Writing effective tools for AI agents" (Anthropic Engineering, 2025): tools are "a contract between deterministic systems and non-deterministic agents." Choose **high-leverage** tools (not thin API wrappers), **namespace** them, return **meaningful, human-readable context**, keep responses **under ~25k tokens**, expose `response_format: concise|detailed`. The context-engineering companion adds the **just-in-time retrieval** pattern: lightweight identifiers, detail on demand.

→ DocxEngine: `docx_*` namespace, summary-not-IDs responses, the 25k cap as an invariant, outline→window reads.

## 3. OOXML is hard for structural reasons

Confirmed from ECMA-376/ISO 29500 and practitioner reports: the split-run problem, rsid fragmentation, the six-layer style cascade, numbering indirection, multi-part stories, and render-time fields. Full treatment: [docs/core/ooxml-pitfalls.md](../core/ooxml-pitfalls.md).

→ DocxEngine: run-coalescing find/replace, rsid stripping, cascade/numbering resolvers, part manager, explicit field-code semantics.

## 4. Every existing library has a disqualifying gap

python-docx (no tracked changes — issue #340 open since Dec 2016; wrong text on revised docs #566), docx-js (generation-focused), docxtemplater (template-bound, paid modules), Pandoc (lossy round-trips, #9833), LibreOffice headless (heavyweight, ~3s+/doc), docx4j/OpenXML SDK (JVM/.NET weight, manual split-run handling), Office.js (host-only). Full comparison: [competitive-landscape.md](competitive-landscape.md).

→ DocxEngine: direct-XML core; wrappers used at most for narrow create paths.

## 5. AI-native approaches validate direct XML — and expose its rough edges

- **Anthropic docx skill**: unzip → grep → patch XML → repack → validate → render-to-JPEG. Powerful but token-heavy and brittle (re-grep line numbers after every script run, manual RSID management, a ~600-line mandatory reference read).
- **SecurityRonin/docx-mcp**: direct-XML MCP server producing real redlines/comments/footnotes; validates every edit against OOXML before saving, catching what "would otherwise cause Word to 'repair'."
- **pablospe/docx-editor**: the addressing innovation — hash-anchored paragraph refs (`P2#f3c1`) with atomic upfront-validated `batch_edit()`.
- **UseJunior/safe-docx**: stable 12-char hex IDs "derived from intrinsic w14:paraId or normalized text — byte-identical across reopens."

→ DocxEngine combines all four lessons in one engine, replacing grep-and-line-numbers with anchors and making validation always-on.

## 6. Structured-format generation is a measurable failure mode

LLMs are measurably worse at XML than JSON (constrained-generation tooling gap; format-restriction effects, arXiv:2408.02442). **PPTArena/PPTPilot** (the PowerPoint analog) found "no single editing modality is sufficient" and won with deterministic OOXML patching inside a plan-edit-check loop. **ParseBench** defines content faithfulness as the fundamental requirement. A consulting-work benchmark found agents fabricating values when files were hard to read — robust read tools are a safety property.

→ DocxEngine: agents emit small JSON tool calls (never XML), the engine does deterministic patching, the check step is built in.

## 7. Stable addressing: raw IDs are untrustworthy

Per [MS-DOCX] §2.6.2.3, `w14:paraId` is unique-within-part — but it's a Word-extension absent from other producers, randomly generated, and **observed being wholesale regenerated by Word on save** (Open-XML-SDK #925, #962). The convergent solution across pablospe/docx-editor and safe-docx: content-hash anchors validated before every edit.

→ DocxEngine: `P{index}#{hash}` with `w14:paraId` demoted to a hint/seed. Full design: [docs/core/anchors.md](../core/anchors.md).

## Caveats carried forward

- Performance/competitor figures from vendor docs and issue trackers should be re-verified under our own workloads (tracked in the benchmark).
- PPTArena/ParseBench numbers are from recent preprints — directionally trusted, not numerically.
- The Anthropic docx skill is proprietary-licensed: approach studied, code never copied ([LICENSING.md](../../LICENSING.md)).
